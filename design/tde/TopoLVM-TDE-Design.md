# Transparent Data Encryption for TopoLVM (LUKS + KMS/Vault/PKCS#11)

**Status:** Draft for review
**Scope:** Add real transparent data encryption (TDE) to the `cloudnativestorage/topolvm` fork (`online-snapshot` branch), using LUKS2/dm-crypt for at-rest encryption and an external key management system (HashiCorp Vault, cloud KMS, or PKCS#11/HSM) for key custody, key rotation, and data re-encryption.
**Inspiration:** `Algonomia/lukscryptwalker-csi` (LUKS-on-local-storage CSI driver). We borrow its node-side LUKS mechanics and Kubernetes-secret passphrase pattern, but replace static passphrases with envelope encryption backed by an external KMS, and we integrate natively into TopoLVM's LVM lifecycle rather than backing files.

---

## 1. Goals and non-goals

### Goals

1. Encrypt every TopoLVM volume at rest with LUKS2/dm-crypt (AES-256-XTS by default), transparently to the workload.
2. Never persist a usable encryption key in plaintext anywhere on disk or in etcd. Keys exist in plaintext only inside the node kernel keyring while a volume is open.
3. Pluggable key custody: HashiCorp Vault (Transit), cloud KMS (AWS KMS, GCP Cloud KMS, Azure Key Vault), and PKCS#11/HSM, behind one provider interface.
4. Key rotation at two levels with zero or minimal downtime: KEK rotation (cheap rewrap) and LUKS keyslot rotation (cheap, online).
5. Online data re-encryption (master-key rotation) for compromise recovery and crypto-agility, using LUKS2 online reencrypt, orchestrated and throttled cluster-wide.
6. Correct interaction with the fork's online snapshots and with clone/restore, expansion, and the existing thin/thick provisioning paths.
7. Auditable: every key, every rotation, every re-encryption is a reconcilable, observable Kubernetes object.

### Non-goals

1. Protecting data from a fully compromised node while the volume is mounted (plaintext is necessarily in kernel memory). TDE protects stolen disks, decommissioned media, raw-LV backups, and offline etcd.
2. Per-file or per-row encryption (that belongs to the application/database layer).
3. In-place conversion of pre-existing plaintext LVs as the primary path (offered as a secondary, best-effort tool in Phase 5).
4. Replacing LVM. We encrypt on top of the LV that lvmd already creates.

## 2. Background: where encryption fits in TopoLVM

TopoLVM's relevant components:

- **topolvm-controller** (Deployment): implements the CSI Controller service (`CreateVolume`, `DeleteVolume`, `CreateSnapshot`, `ControllerExpandVolume`, ...). It does not touch disks directly. It creates and reconciles `LogicalVolume` custom resources.
- **topolvm-node** (DaemonSet): implements the CSI Node service (`NodeStageVolume`, `NodePublishVolume`, `NodeExpandVolume`, `NodeUnstageVolume`). It watches `LogicalVolume` CRs bound to its node and drives lvmd. It owns `mkfs` and mount.
- **lvmd** (gRPC service, host systemd unit or DaemonSet): the only component that runs LVM commands (`lvcreate`, `lvremove`, `lvextend`, thin snapshot creation). Exposes device classes mapped to volume groups.
- **LogicalVolume CRD**: the bridge between controller (desired state) and node/lvmd (actual LV).
- CSI sidecars: external-provisioner, external-resizer, external-snapshotter, node-driver-registrar, livenessprobe.

The raw block device lvmd produces (for example `/dev/topolvm/<uuid>`) is the natural place to insert LUKS. The encryption layer sits **between the LV (ciphertext) and the filesystem (plaintext)**:

```
workload
   |  (plaintext I/O)
filesystem (ext4/xfs)         <- mkfs happens here, on the mapper device
   |
/dev/mapper/topolvm-<id>      <- dm-crypt target (plaintext view)
   |  (LUKS2 / dm-crypt, master key in kernel keyring)
/dev/topolvm/<uuid>           <- the LVM logical volume (ciphertext + LUKS2 header)
   |
LVM thin/thick pool  ->  VG  ->  physical disk
```

Snapshots, expansion, and re-encryption all operate on the **ciphertext device** (`/dev/topolvm/<uuid>`), never on the plaintext mapper device. This single rule keeps the design consistent.

## 3. Key hierarchy (envelope encryption)

Three layers, classic envelope model:

1. **Data** is encrypted by dm-crypt using the **LUKS master key** (a.k.a. volume key). This key is generated by `cryptsetup luksFormat` and never leaves the LUKS header except into the kernel when the device is opened. Rotating it means rewriting all data (re-encryption).
2. The master key is stored in a **LUKS2 keyslot**, wrapped by a key derived (Argon2id) from a **per-volume passphrase** (the DEK in our terms). Adding/removing a passphrase is a keyslot operation and does not touch data.
3. The per-volume passphrase (DEK) is itself stored only in **wrapped (ciphertext) form**, wrapped by a **KEK** that lives in the external KMS/HSM and never leaves it. Unwrapping requires an authenticated call to the provider.

```
data  <--dm-crypt(master key)--  LUKS master key
LUKS master key  <--Argon2id(passphrase)--  keyslot in LUKS2 header (on the LV)
passphrase (DEK)  <--KMS/HSM wrap(KEK)-->  wrapped DEK blob (stored in etcd, ciphertext only)
KEK  ---- never leaves Vault/KMS/HSM
```

Consequences that drive the rest of the design:

- **Unlock at mount** = one provider `Unwrap` call to recover the passphrase, then `cryptsetup luksOpen`. Provider outage means volumes cannot be opened (mitigations in 8.4).
- **KEK rotation** = re-wrap the per-volume passphrase with the new KEK version. No disk I/O, no downtime. (Vault Transit `rewrap`, KMS `ReEncrypt`.)
- **Passphrase rotation** = `luksAddKey` a new passphrase into a free keyslot, then `luksKillSlot` the old one. Online, no data rewrite, but it changes the header, which matters for already-taken snapshots (Section 7).
- **Master-key rotation** = `cryptsetup reencrypt`, the only operation that actually rewrites ciphertext (Section 9).

## 4. Provider abstraction

A single Go interface, one implementation per backend. The driver speaks only to this interface; cryptsetup only ever sees a passphrase.

```go
// KeyProvider performs envelope operations. The KEK never leaves the provider.
type KeyProvider interface {
    // GenerateDEK returns a fresh high-entropy per-volume secret (the LUKS passphrase),
    // already wrapped by the provider's current KEK. Plaintext is returned once,
    // held in a memory-locked buffer, and zeroized by the caller after luksFormat.
    GenerateDEK(ctx context.Context, opts KeyOpts) (plaintext SecretBuf, wrapped WrappedKey, err error)

    // Unwrap recovers the plaintext passphrase from a wrapped blob for luksOpen.
    Unwrap(ctx context.Context, wrapped WrappedKey) (plaintext SecretBuf, err error)

    // Rewrap re-encrypts the wrapped blob under the provider's current KEK version
    // without exposing plaintext to the driver (Vault Transit rewrap / KMS ReEncrypt).
    // Used for KEK rotation. Returns the new blob and KEK version.
    Rewrap(ctx context.Context, wrapped WrappedKey) (WrappedKey, error)

    // KEKVersion reports the provider's current key version (for drift detection).
    KEKVersion(ctx context.Context, keyRef string) (string, error)
}

type WrappedKey struct {
    Ciphertext []byte // wrapped passphrase, safe to store in etcd
    KeyRef     string // provider key id: vault transit key, KMS key ARN, PKCS#11 label
    KEKVersion string // version used to wrap, drives rewrap reconciliation
    Provider   string // "vault" | "aws-kms" | "gcp-kms" | "azure-kv" | "pkcs11"
}
```

### Provider implementations

- **Vault (Transit)**: `GenerateDEK` = `transit/datakey/plaintext/<key>` (Vault returns plaintext + ciphertext of a data key); `Unwrap` = `transit/decrypt`; `Rewrap` = `transit/rewrap`. KEK rotation = `transit/keys/<key>/rotate` then reconcile blobs to the new version. Auth via Kubernetes auth method (the node/controller service account JWT). This is the recommended default: it gives true rewrap without ever exposing the KEK, and native rotation.
- **AWS KMS**: `GenerateDEK` = `GenerateDataKey`; `Unwrap` = `Decrypt`; `Rewrap` = `ReEncrypt`. IRSA for auth. Encryption context = `{volumeID, namespace}` for tamper-evident binding.
- **GCP Cloud KMS / Azure Key Vault**: same envelope pattern (`Encrypt`/`Decrypt`, key versions for rotation). Workload Identity / Managed Identity for auth.
- **PKCS#11 / HSM**: wrap/unwrap the passphrase with an HSM-resident AES or RSA key via a PKCS#11 session. The HSM key is the KEK and is non-exportable. `Rewrap` unwraps under the old HSM key handle and wraps under the new one inside the token. Useful where compliance requires hardware key custody. (We deliberately do not push unlocking into LUKS2 PKCS#11 tokens; keeping wrap/unwrap in the driver gives one uniform model across providers.)

### Auth posture

- Controller-side provider calls (provisioning, rewrap) use the controller service account.
- Node-side provider calls (unwrap at mount, reencrypt) use the node service account, scoped to only `Unwrap`/`Decrypt` plus, for re-encryption, `GenerateDEK`. Nodes should not hold KEK-rotation rights.
- Encryption context / Vault key policy binds each wrapped blob to its `volumeID` so a blob cannot be replayed against another volume.

## 5. API surface

### 5.1 StorageClass parameters

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: topolvm-encrypted
provisioner: topolvm.io
parameters:
  topolvm.io/device-class: "ssd"

  topolvm.io/encryption: "true"
  topolvm.io/key-provider: "vault"              # vault | aws-kms | gcp-kms | azure-kv | pkcs11
  topolvm.io/key-ref: "transit/keys/topolvm"    # provider-specific KEK identifier
  topolvm.io/cipher: "aes-xts-plain64"          # default
  topolvm.io/key-size: "512"                    # bits; 512 = AES-256-XTS
  topolvm.io/pbkdf: "argon2id"
  topolvm.io/integrity: "none"                  # none | hmac-sha256 (Phase 5, perf cost)
  topolvm.io/discards: "false"                  # allow TRIM passthrough (leaks usage pattern)
  csi.storage.k8s.io/fstype: "ext4"
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Delete
```

Provider credentials/roles are referenced by name (Vault role, IRSA annotation, PKCS#11 token label) rather than embedded; no long-lived secret material in the StorageClass.

### 5.2 LogicalVolume CRD extension

Additive fields, backward compatible:

```yaml
spec:
  # ...existing fields...
  encryption:
    enabled: true
    provider: vault
    keyRef: transit/keys/topolvm
    cipher: aes-xts-plain64
    keySize: 512
status:
  # ...existing fields...
  encryption:
    state: Opened            # Pending | Formatted | Opened | Reencrypting | Error
    headerUUID: "a1b2..."    # LUKS2 header UUID, identifies the on-disk header
    activeKeyID: "vk-7f3c"   # which EncryptionKey object currently unlocks this volume
    masterKeyEpoch: 2        # bumped only by a completed reencrypt
```

### 5.3 New CRD: EncryptionKey

One per logical key lineage (a volume's key, or a snapshot's pinned key). Holds **only ciphertext**.

```yaml
apiVersion: topolvm.io/v1
kind: EncryptionKey
metadata:
  name: vk-7f3c
spec:
  provider: vault
  keyRef: transit/keys/topolvm
status:
  wrappedDEK: "<base64 ciphertext>"   # passphrase wrapped by KEK; never plaintext
  kekVersion: "v4"                     # current wrap version, reconciled toward provider current
  boundVolumeID: "pvc-1234"            # encryption context binding
  keyslot: 0                           # LUKS2 keyslot this passphrase occupies
  consumers: ["pvc-1234"]              # volumes/snapshots that depend on this key
  createdAt: ...
  lastRewrapAt: ...
  retiredAt: null                      # set when superseded but still needed by a snapshot
```

The wrapped DEK could alternatively live in a `Secret`; we prefer a CRD for status, audit, finalizers, and reconciliation. etcd-at-rest encryption (KMS provider for the API server) is still recommended as defense in depth, but security does not depend on it because the blob is already ciphertext.

### 5.4 New CRD: ReencryptRequest

Drives master-key rotation as an auditable, throttled operation (Section 9).

```yaml
apiVersion: topolvm.io/v1
kind: ReencryptRequest
metadata:
  name: rotate-2026q3
spec:
  selector:                      # which volumes
    matchLabels: {compliance: pci}
  reason: "annual-key-rotation"
  newCipher: aes-xts-plain64     # optional crypto-agility upgrade
  maxConcurrentPerNode: 1
  throughputLimit: "50MiB/s"
status:
  phase: Running                 # Pending | Running | Completed | Failed
  total: 120
  completed: 87
  failed: 0
  perVolume: [...]
```

## 6. Volume lifecycle (node flows)

The node container ships `cryptsetup` and runs LUKS operations in the host mount/device namespace (via `nsenter` into the host, as TopoLVM already does for mounts), privileged. Passphrases are passed to cryptsetup over **stdin** (`--key-file=-`), never via argv or temp files, and held in `mlock`ed buffers, zeroized immediately after use.

### 6.1 Provision + first mount

1. **CreateVolume (controller)**: as today, create the `LogicalVolume` CR. Additionally, if encryption is enabled, call `provider.GenerateDEK`, create an `EncryptionKey` object with the wrapped DEK, and set `lv.spec.encryption` + `status.encryption.activeKeyID`. The plaintext passphrase is **not** sent to the controller's disk or logs; only the wrapped blob is persisted.
2. lvmd creates the raw LV exactly as today (thin or thick). It is plaintext-uninitialized, no LUKS header yet.
3. **NodeStageVolume (node)**:
   a. Read `lv.status.encryption`; resolve the `EncryptionKey`; `provider.Unwrap` -> passphrase in a locked buffer.
   b. `cryptsetup isLuks /dev/topolvm/<uuid>`?
      - **No** (fresh LV): `cryptsetup luksFormat --type luks2 --cipher aes-xts-plain64 --key-size 512 --pbkdf argon2id /dev/topolvm/<uuid> -` (passphrase on stdin). Record `headerUUID`, `keyslot=0`.
      - **Yes** (re-stage of an existing volume): skip format.
   c. `cryptsetup open /dev/topolvm/<uuid> topolvm-<id> -` -> `/dev/mapper/topolvm-<id>`.
   d. First time only: `mkfs.<fstype>` on the mapper device.
   e. Mount the mapper device at the staging path. Apply `fsGroup`/permissions as TopoLVM already does.
   f. Zeroize the passphrase buffer. The master key now lives only in the kernel device-mapper context.
4. **NodePublishVolume**: bind-mount staging path into the pod (unchanged). For `volumeMode: Block`, publish the mapper device directly.

### 6.2 Unmount

**NodeUnstageVolume**: unmount, `cryptsetup close topolvm-<id>` (drops the master key from the kernel), revoke any keyring entry. The LV is ciphertext at rest again.

### 6.3 Expansion

**NodeExpandVolume**, after lvmd `lvextend` grows the ciphertext LV:
1. `cryptsetup resize topolvm-<id>` (grows the dm-crypt mapping; needs the key, so unwrap again or use the already-open device's key).
2. Grow the filesystem (`resize2fs` / `xfs_growfs`) on the mapper device.

Order matters: extend LV, then resize crypt, then grow fs. Shrink is not supported (consistent with TopoLVM).

### 6.4 Delete

**DeleteVolume**: lvmd removes the LV (ciphertext and header gone). The controller deletes the `EncryptionKey` **unless** a snapshot still pins it (finalizer + `consumers` list). Crypto-erase is effectively immediate once both the LV header and the wrapped DEK are gone, even if raw blocks linger.

## 7. Snapshots and clones (the online-snapshot branch)

The fork takes LVM thin snapshots of mounted volumes, using a filesystem freeze for crash consistency. Encryption changes two things and we must get both right.

### 7.1 Freeze the plaintext filesystem, snapshot the ciphertext LV

The freeze must target the **filesystem on `/dev/mapper/topolvm-<id>`** (FIFREEZE), then the LVM thin snapshot is taken on the **underlying ciphertext LV `/dev/topolvm/<uuid>`**, then thaw. This yields a crash-consistent ciphertext snapshot. We never snapshot the mapper device. The node code that performs the freeze must resolve mapper -> underlying LV (already known from staging state).

### 7.2 A thin snapshot inherits the origin's LUKS header

An LVM thin snapshot is a block-level copy-on-write of the ciphertext, so the snapshot **contains the same LUKS2 header, same master key, and the same keyslots that existed at snapshot time**. Therefore:

- The snapshot is openable with whatever passphrase unlocked the origin **at the moment of the snapshot**.
- We **pin** the key: at `CreateSnapshot`, copy the origin's current `EncryptionKey` (wrapped DEK + keyslot) into a new `EncryptionKey` object owned by the snapshot, and add the snapshot to its `consumers`. The origin's `EncryptionKey` gets a finalizer so passphrase rotation or deletion cannot strand the snapshot.
- This is why passphrase rotation (Section 8.2) and master-key reencrypt (Section 9) must record key epochs: a snapshot taken before a rotation needs its pinned (older) key, which the origin's live header may no longer accept.

### 7.3 Restore / clone (CreateVolume from snapshot)

1. lvmd creates a new thin LV from the snapshot (block copy) -> it carries the snapshot's header and master key.
2. The restored volume initially uses the snapshot's pinned `EncryptionKey`.
3. **Re-key on divergence (recommended):** immediately after restore, give the clone its own identity: `provider.GenerateDEK` -> `luksAddKey` (new keyslot) -> `luksKillSlot` (old) -> new `EncryptionKey`. Now the clone and origin/snapshot no longer share an unlockable passphrase, even though they briefly shared a master key. For full cryptographic separation (different master key), schedule a `ReencryptRequest` on the clone; document this as the high-assurance, higher-cost option.

## 8. Key rotation (no data rewrite)

Two cheap rotations, both online. Reconciled by a **key controller** in topolvm-controller.

### 8.1 KEK rotation (rewrap)

Trigger: scheduled policy, or detection that the provider's current KEK version is newer than an `EncryptionKey`'s `kekVersion`.

Action: `provider.Rewrap(wrapped)` -> store the new ciphertext + version in the `EncryptionKey`. No node involvement, no volume open, no I/O. Vault Transit `rewrap` and KMS `ReEncrypt` perform this without exposing plaintext. This is the routine, frequent rotation and should be fully automated (for example reconcile all blobs to the provider's current version daily).

### 8.2 LUKS passphrase / keyslot rotation

Trigger: policy (rotate per-volume passphrases every N days), or operator request.

Action (node, on an open or openable volume):
1. `provider.GenerateDEK` -> new passphrase + wrapped blob.
2. `cryptsetup luksAddKey /dev/topolvm/<uuid> --key-file <old-on-stdin>` adding the new passphrase to a free keyslot.
3. `cryptsetup luksKillSlot` the old keyslot.
4. Replace the volume's `EncryptionKey` (mark the old one `retiredAt`; keep it only if a snapshot pins it).

This re-wraps the master key under a new passphrase without rewriting data. It is online and fast. It does **not** change the master key, so it does not help if the master key itself is suspected compromised; that is re-encryption (Section 9). Snapshots taken before this rotation keep their pinned older keyslot copy (Section 7.2).

## 9. Data re-encryption (master-key rotation)

The only operation that rewrites ciphertext. Needed for: suspected master-key compromise, regulatory hard-rotation, or cipher upgrade (crypto-agility).

### 9.1 Mechanism

LUKS2 supports **online reencryption** while the device is mounted:

```
cryptsetup reencrypt /dev/topolvm/<uuid> \
  --resilience checksum \         # crash-safe; can resume after node reboot/crash
  --key-file <current-on-stdin>
# optionally --cipher <new> for crypto-agility
```

`--resilience checksum` (or `journal`) makes the operation restartable after an interruption, which is essential because a volume can be large and a node can reboot mid-operation. The node performs the reencrypt on the ciphertext device while the filesystem stays mounted on the mapper device.

### 9.2 Orchestration

A `ReencryptRequest` (Section 5.4) is reconciled by a **reencryption controller**:

1. Expand the selector to a concrete volume list.
2. Respect `maxConcurrentPerNode` (default 1) and a global cap, so re-encryption never saturates node I/O. Apply `ionice` / `--reencrypt` hotzone sizing and the `throughputLimit`.
3. For each volume, signal the owning node (via a per-LV annotation or a `LogicalVolume.status.encryption.state = Reencrypting`) to run `cryptsetup reencrypt`.
4. On completion: bump `status.encryption.masterKeyEpoch`, generate a fresh `EncryptionKey` if the passphrase also changed, mark progress.
5. Surface progress in `status` and metrics; make the request resumable (a crashed controller or node picks up where LUKS left off, since LUKS itself tracks reencrypt progress in the header).

### 9.3 Interaction with snapshots

A re-encrypted origin gets a **new master key**. Pre-existing snapshots are independent block copies that still carry the **old** master key and are still readable via their pinned `EncryptionKey`. We therefore:

- Keep retired `EncryptionKey` objects (and their wrapped DEKs) alive as long as any snapshot pins them (finalizer + `consumers`).
- Garbage-collect a retired key only when its last consuming snapshot is deleted.

New snapshots taken after the reencrypt capture the new master key, as expected.

### 9.4 Cost and guidance

Re-encryption reads and writes every block, so it is I/O-bound and proportional to volume size. Guidance baked into defaults: one volume per node at a time, throttled, off-peak windows via the request schedule, and always `--resilience checksum`. KEK rotation (8.1) and keyslot rotation (8.2) cover the vast majority of rotation needs cheaply; reserve full re-encryption for compromise and compliance hard-rotation.

## 10. Threat model

**Protects against:**
- Theft or improper disposal of a physical disk or SSD.
- Backups/exports of the raw LV or VG (ciphertext only).
- Offline compromise of etcd (only wrapped DEKs and LUKS headers are stored; both useless without the KMS/HSM KEK).
- A decommissioned node whose volumes were closed (no master key resident).

**Does not protect against:**
- Root on a node while a volume is open (plaintext is in kernel memory and the mounted fs is readable). Mitigate with node hardening, RBAC on the driver's secrets, and minimizing privileged access.
- A compromised KMS/HSM or stolen provider credentials (the KEK is the root of trust). Mitigate with short-lived workload-identity auth, encryption-context binding, audit logging, and HSM-backed non-exportable KEKs.
- Application-layer leakage (logs, swap). Recommend encrypted swap or `swapoff` on nodes, and `mlock` for the driver (already required).

**Key-handling hygiene (driver):**
- Passphrases only via stdin to cryptsetup; never argv, env, or temp files.
- `mlock` all secret buffers; explicit zeroization after use; `GODEBUG` and core-dump suppression on the node binary.
- Master key lives only in the kernel device-mapper context between `open` and `close`.
- No plaintext key ever reaches controller logs, node logs, or CR status.

## 12. Testing strategy

- **csi-sanity** against the encrypted StorageClass to confirm CSI conformance is preserved.
- **e2e on kind** with a loopback-backed VG: provision, write, unmount/remount (proves data survives a close/open cycle), expand, snapshot, restore, delete.
- **Crash/fault injection:** kill the node mid-`luksFormat`, mid-`open`, and mid-`reencrypt`; confirm `--resilience checksum` resumes and no plaintext leaks.
- **Snapshot correctness:** snapshot under active writes (with freeze), restore, verify the restored fs is consistent and opens with the pinned key; verify a post-rotation origin and a pre-rotation snapshot are both readable with their respective keys.
- **Provider failure modes:** Vault sealed / KMS throttled / HSM unreachable at mount time -> volume fails to open with a clear, non-leaky error and retries; at provision time -> CreateVolume fails cleanly.
- **Rotation:** automated KEK rewrap reconciles all blobs to the new version; keyslot rotation leaves data intact; full reencrypt changes `masterKeyEpoch` and the old master key still opens retained snapshots.
- **Performance:** fio baseline vs encrypted (AES-NI on), with and without `--perf-no_read_workqueue/--perf-no_write_workqueue`; reencrypt throughput under the throttle.
- **Security regression:** scan logs and process args in CI to assert no plaintext passphrase or key material is ever emitted.

## 13. Open questions

1. **lvmd ownership of cryptsetup vs node:** we place LUKS in topolvm-node (keeps lvmd a pure LVM service). Confirm this fits the fork's process boundaries and the host-namespace execution model.
2. **EncryptionKey vs Secret:** CRD chosen for audit/finalizers; confirm acceptable given it adds a watch and finalizer surface.
3. **dm-integrity:** worth offering in Phase 5 given its performance cost and its interaction with thin snapshots (integrity metadata enlarges the device and complicates resize). Default stays confidentiality-only XTS.
4. **Discards/TRIM:** default off (avoids leaking allocation patterns); expose per-StorageClass for users who need TRIM on SSDs.
5. **Per-node concurrency for reencrypt** under heavy multi-tenant nodes: is a single global throttle plus per-node cap sufficient, or do we need priority classes per `ReencryptRequest`?
6. **FIPS / cipher policy:** whether to pin Argon2id parameters and cipher set centrally (a cluster `EncryptionPolicy` CRD) rather than per-StorageClass.
